Architecture

One daemon, many clients

adlrd is the single authority over all sessions and agent processes. The CLI and TUI are thin clients — they send commands or subscribe to events over a Unix socket. Persistence is hidden behind a Storage interface, making the backend swappable.

Daemon lifecycle

socket ~/.local/share/adlr/adlr.sock — one global daemon for all projects
start CLI auto-spawns detached Bun process if socket not found; polls 100ms until ready
stop Graceful SIGTERM — waits for running agents. Auto-exits after 10 min inactivity.
Data Model

Spans and events, all the way down

Instead of separate tables for agents, workflow steps, and hooks, adlr uses a single spans table for every unit of work. A kind field says what it is. Adding new kinds — workflow steps, hooks, assistant runs — costs nothing: no schema change, no migration.

Everything that happens is an event with an open-ended type string and a JSON payload. This is the OpenTelemetry model applied to agent orchestration.

Span context propagation

Context flows through environment variables — the subprocess equivalent of W3C Trace Context over HTTP headers. When the daemon spawns an agent it injects the span context. If that agent calls adlr agent run, the CLI reads ADLR_SPAN_ID from its own environment and passes it as the parent. Propagation works at any depth automatically.

ADLR_SESSION=session-abc123
ADLR_SPAN_ID=span-xyz789  ← this agent's span id
ADLR_SOCKET=~/.local/share/adlr/adlr.sock
ADLR_AGENT_PROMPT="Implement the payment module"
ADLR_CONTEXT=[{"type":"goal","value":{"text":"..."}}, ...]

Agent Execution

What happens when you run an agent

Every adlr agent run follows the same flow. The daemon is the authority throughout — spawning, tracking, streaming output, and broadcasting to any connected TUI clients in real time.

1

CLI connects

SDK connects to socket. Auto-starts daemon if not running.

2

Span created

Daemon creates a span (kind=agent) in SQLite via Storage interface.

3

PTY spawn

Daemon calls Bun.spawn() with PTY. Injects ADLR_* env vars.

4

Output streams

Each stdout line → agent.output event in DB → pushed to TUI.

5

Span closes

On exit: span updated to done/failed, event emitted, TUI re-renders.

CLI

Control everything from the terminal

All commands are thin wrappers around SDK calls. Session context is resolved automatically from --session flag, ADLR_SESSION env, or .adlr/.session file.

Sessions
adlr Open TUI dashboard for current session
adlr new [--goal "..."] Create session, write id to .adlr/.session
adlr session list List all sessions across all projects
adlr init Scaffold .adlr/adlr.ts in current project
Agents
adlr agent run --agent <type> <prompt> Spawn agent, returns span id
adlr agent wait [--name <n>] Block until span is done or failed
adlr agent status [--name <n>] Print current span status
adlr agent read [--name <n>] Stream agent PTY output to stdout
adlr agent list List all agent spans for session
Context
adlr context add --type <t> <value> Add context item (with optional --label, --description)
adlr context list List all context items for session
adlr context get [--type t] [--label l] Filtered output — used in workflow step prompts
Daemon
adlr daemon stop Graceful shutdown — waits for running agents
Configuration

Your stack, your config

adlr is configured with a TypeScript file. Global config at ~/.config/adlr/adlr.ts, project config at .adlr/adlr.ts. Project config merges over global. Agents are just functions returning shell commands.

import type { AdlrConfig } from "@adlr/sdk"

const config: AdlrConfig = {
  agent: {
    agents: {
      // Any CLI agent — return a shell command string
      opencode: ({ prompt, subagent }) =>
        `opencode run --agent ${subagent} "${prompt}"`,
    },
    // Invoked when pressing 'o' on an agent in the TUI
    attach: ({ agentId }) =>
      `tmux new-window "adlr agent read --name ${agentId}"`,
  },
}

export default config
Roadmap

Four phases to a complete platform

Each phase produces a working, useful system. The span and event data model is designed so that Phases 3 and 4 add capabilities without touching the schema.

Phase 1 + 2 — Now

Core + TUI

The foundation. A global daemon managing sessions and agents. Full SQLite-backed span/event tracing. SDK, CLI, and a live Ink dashboard with five tabs — overview, context, agents, traces, and logs.

adlrd daemon @adlr/sdk CLI Ink TUI SQLite PTY attach
Phase 3 — Next

Workflows, Hooks & Plugins

YAML or TypeScript workflow definitions with sequential and parallel steps. Before/after hooks for any adlr event. npm plugin system for distributing agent configs — first-party plugin: @adlr/opencode.

adlr run Workflow engine Hooks @adlr/opencode
Phase 4 — Planned

adlr Assistant

An AI agent with full awareness of the current session — all spans, events, and context. Ask it anything about the session, or let it auto-orchestrate: read state, decide what to run next, iterate until done.

adlr assistant --auto mode Session-aware AI
Future

Web UI & Remote Sessions

A browser dashboard built on @adlr/sdk — just another client. Remote daemon support via TCP socket allows connecting from anywhere. Multi-user sessions for team visibility.

Web dashboard Remote daemon Multi-user
