adlr — Revised Data Model

Section 2 of 4 (revised) — Span-based observability model

SESSION
id TEXT PK
goal TEXT
status TEXT  active|completed|archived
working_dir TEXT
created_at INTEGER
SPAN  — every unit of work
id TEXT PK
session_id TEXT FK
parent_id TEXT FK? ← nesting
kind TEXT
agent | workflow | step | hook | ...
name TEXT
status TEXT
pending|running|done|failed
started_at INTEGER
finished_at INTEGER
data JSON ← kind-specific fields
agent: {prompt, agent_type, pid, exit_code}
step: {workflow, step_name, index}
EVENT  — everything that happened
id INTEGER PK
session_id TEXT FK
span_id TEXT FK? ← nullable
type TEXT ← open-ended
agent.output | log.info | log.warn
context.added | span.started | ...
data JSON
timestamp INTEGER
CONTEXT_ITEM  — unchanged
id TEXT PK
session_id TEXT FK
type TEXT   url | file | text
label TEXT
description TEXT
value TEXT
created_at INTEGER

Example span tree

session: "Implement payment feature"
└─ span: workflow:payment [kind=workflow]
├─ span: step:research [step]
└─ span: opencode:plan [agent]
├─ event: agent.output "Reading..."
└─ event: agent.output "Done"
└─ span: step:implement [step]
├─ span: agent-1 ← parallel
└─ span: agent-2 ← parallel

What this enables

  • Add workflows in Phase 3 — zero schema change
  • Add hooks in Phase 3 — just a new span kind
  • Parallel agents visible as sibling spans
  • TUI shows a live span tree, not just a flat list
  • New event types added freely — no migration
  • Full replay: reconstruct any session from events
Storage interface still applies: the daemon accesses all four tables through a Storage abstraction. SQLite is one implementation — anything can be swapped in behind it.
